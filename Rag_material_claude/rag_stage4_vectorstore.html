How Vector Databases Work START HERE
What makes vector databases fundamentally different from SQL/NoSQL — and why you need one for RAG.
One-line definition: A vector database stores embeddings (float vectors) alongside their original text and metadata, and lets you find the most semantically similar vectors to a query in milliseconds — even among billions of records.

Traditional DB vs Vector DB — the key difference

🗄️ Traditional SQL/NoSQL
Query: "Find rows WHERE category = 'AI'"
Match: exact match only
"AI research" ≠ "artificial intelligence"

Works great for structured data, fails for semantic meaning.
🔮 Vector Database
Query: embed("What is artificial intelligence?")
Match: semantic similarity
"AI research" ✓ "machine learning" ✓ "deep learning" ✓

Finds meaning, not just exact keywords.

What is stored inside a vector database record?

what_is_stored.py — anatomy of a vector DB record
# Every record in a vector database has THREE components:

record = {
    # 1. THE VECTOR — the float embedding (the actual searchable data)
    "embedding": [0.021, -0.134, 0.876, 0.003, ...],  # 384–3072 floats

    # 2. THE DOCUMENT — the original text (what gets returned to LLM)
    "document": "Retrieval Augmented Generation (RAG) is a technique...",

    # 3. THE METADATA — structured info for filtering (optional but powerful)
    "metadata": {
        "source":     "docs/rag_guide.pdf",
        "page":       3,
        "data_type":  "pdf",
        "chapter":    "Introduction",
        "date":       "2025-01-15",
        "language":   "en",
        "word_count": 142,
    }
}

# When you query the database:
# 1. Your question gets embedded → query_vector
# 2. Database finds top-k records whose embedding is closest to query_vector
# 3. Returns document + metadata (NOT the raw vectors)
# 4. You pass documents to LLM as context → final answer

# Key insight: you NEVER manually compare vectors yourself.
# The vector DB does it for you, in milliseconds, for millions of records.

The RAG retrieval loop — what happens at query time

1
User asks a question

"What are the side effects of aspirin?"

2
Question gets embedded

Same embedding model used to build the index → query vector [0.12, -0.34, ...]

3
Vector DB finds nearest neighbours

HNSW graph search — finds top-5 most similar chunk vectors in milliseconds.

4
Documents returned

The text content of those 5 chunks is returned — not the vectors themselves.

5
LLM generates the answer

Prompt = "Answer this using only this context: [5 chunks] Question: [user question]"

Internal Architecture DEEP DIVE
How vectors are actually stored, indexed, and searched inside the database. Understanding this helps you tune performance and choose the right database.
The core problem: With 1 million 768-dim vectors, comparing a query to every vector (brute force) = 1 million distance calculations. At 10M vectors, that's 10 million. Too slow for real-time use. Vector databases solve this with indexing algorithms — ANN (Approximate Nearest Neighbour) search.

ANN Index Types — the algorithms inside vector databases

AlgorithmFull NameHow It WorksSpeedRecallUsed By
HNSWHierarchical Navigable Small WorldMulti-layer graph; navigate from coarse to fine layers⚡⚡⚡⚡99%+All major DBs
IVFInverted File IndexCluster vectors into buckets; search only relevant buckets⚡⚡⚡95%+FAISS, Milvus
FlatBrute-force exact searchCompare query to every vector — 100% recall, slow at scale⚡100%FAISS, small DBs
PQProduct QuantizationCompress vectors → much less RAM; slightly lower recall⚡⚡⚡⚡92%+FAISS (IVFPQ)
DiskANNDisk-based ANNStore index on disk not RAM; handles billions of vectors⚡⚡⚡97%+Weaviate, Milvus

HNSW — the dominant algorithm (used in almost every vector DB)

HNSW builds a hierarchical graph. Search starts at the sparse top layer (fast navigation) and zooms into dense lower layers (precise matching). Think of it like Google Maps — start with continent view, zoom to street level.

Key HNSW parameters you'll tune:
M — max connections per node per layer (16–64). Higher = better recall but more RAM.
ef_construction — how many candidates considered during build (100–200). Higher = better index, slower build.
ef_search — search beam width at query time (50–100). Higher = better recall, slower queries.

Distance Metrics — how "closeness" is measured

distance_metrics.py
import numpy as np

a = np.array([0.1, 0.8, 0.3])  # embedding of chunk
b = np.array([0.2, 0.7, 0.4])  # embedding of query

# ── 1. COSINE SIMILARITY (most common for text) ────────
# Measures the angle between vectors — ignores magnitude
# Range: -1 to 1  |  1 = identical, 0 = unrelated, -1 = opposite
# Use when: all text embeddings (normalized vectors)
cosine = np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
print(f"Cosine similarity: {cosine:.4f}")

# ── 2. EUCLIDEAN DISTANCE (L2) ─────────────────────────
# Straight-line distance in vector space
# Range: 0 to ∞  |  0 = identical, higher = more different
# Use when: image embeddings, spatial data
euclidean = np.linalg.norm(a - b)
print(f"Euclidean distance: {euclidean:.4f}")

# ── 3. DOT PRODUCT ────────────────────────────────────
# Like cosine but also considers vector magnitude
# Fast to compute. Use when: OpenAI embeddings (they recommend it)
dot = np.dot(a, b)
print(f"Dot product: {dot:.4f}")

# ── Which to use in your RAG pipeline? ────────────────
# Text with normalized embeddings (bge, MiniLM, nomic) → COSINE
# OpenAI embeddings → DOT PRODUCT (OpenAI recommends this)
# Image/spatial → EUCLIDEAN
# Rule: if unsure → use COSINE. It works for everything.

What happens physically when you store a vector

1
Vector inserted

The float array is stored in the DB's vector storage layer (RAM or disk depending on DB).

2
HNSW graph updated

The new vector is probabilistically assigned to graph layers and connected to M nearest neighbours per layer.

3
Metadata stored separately

The document text and metadata dict are stored in a standard key-value or document store alongside the vector.

4
ID links everything

A unique ID links the vector index entry → vector array → document → metadata. Everything is retrievable via this ID.

5
Persisted to disk

Most DBs persist the HNSW graph to disk so the index survives restarts without rebuilding.

Metadata — Structure, Significance & Filtering CRITICAL CONCEPT
Metadata is the second most important concept after the vector itself. Without it, you can only search by semantic similarity. With it, you can combine semantic search with structured filtering.
Why metadata matters: "Find the top 5 chunks similar to my query" is basic retrieval. "Find the top 5 chunks similar to my query from PDFs published after 2024, in the 'Introduction' section, in English" — that's metadata-filtered retrieval. It dramatically reduces noise in your RAG context.

Metadata filtering in LangChain — practical code

metadata_filtering.py
from langchain_chroma import Chroma

# Assume vectorstore is already built (see ChromaDB tab)
# ── Basic metadata filter ──────────────────────────────
results = vectorstore.similarity_search(
    query="What is the refund policy?",
    k=5,
    filter={"data_type": "pdf"}   # only search PDF chunks
)

# ── Multi-field filter ─────────────────────────────────
results = vectorstore.similarity_search(
    query="machine learning techniques",
    k=5,
    filter={
        "data_type": "pdf",
        "language": "en"
    }
)

# ── Chroma: $and / $or operators ──────────────────────
results = vectorstore.similarity_search(
    query="neural network training",
    k=5,
    filter={
        "$and": [
            {"data_type": {"$eq": "pdf"}},
            {"page": {"$gte": 5}}    # only pages 5 and above
        ]
    }
)

# Chroma filter operators:
# $eq   → equals           {"field": {"$eq": "value"}}
# $ne   → not equals       {"field": {"$ne": "value"}}
# $gt   → greater than     {"field": {"$gt": 5}}
# $gte  → greater or equal {"field": {"$gte": 5}}
# $lt   → less than        {"field": {"$lt": 100}}
# $lte  → less or equal    {"field": {"$lte": 100}}
# $in   → in list          {"field": {"$in": ["a","b","c"]}}
# $nin  → not in list      {"field": {"$nin": ["x","y"]}}
# $and  → all conditions   {"$and": [{...}, {...}]}
# $or   → any condition    {"$or":  [{...}, {...}]}

# ── Retrieval with score (see similarity score) ────────
results_with_score = vectorstore.similarity_search_with_score(
    query="What are embedding models?",
    k=5
)
for doc, score in results_with_score:
    print(f"Score: {score:.4f} | Source: {doc.metadata['source']}")
    print(f"Text: {doc.page_content[:100]}\n")

# ── Filter by source file (powerful for multi-doc RAG) ─
def search_in_file(vectorstore, query: str, filename: str, k=5):
    return vectorstore.similarity_search(
        query, k=k,
        filter={"source": {"$eq": filename}}
    )

results = search_in_file(vectorstore, "refund rules", "docs/policy.pdf")
Enriching metadata during ingestion (Stage 1): Always add useful metadata at the ingestion stage — date, language, category, word_count. You cannot easily add metadata after embedding. Plan your metadata schema before you build the index.

from langchain_chroma import Chroma
from langchain_huggingface import HuggingFaceEmbeddings
from langchain.schema import Document

# ── 1. Load embedding model ──────────────────────────
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-small-en-v1.5",
    model_kwargs={"device": "cpu"},
    encode_kwargs={"normalize_embeddings": True}
)

# ── 2a. CREATE a new vector store from chunks ─────────
# stage2_chunks = list[Document] from your chunking stage
vectorstore = Chroma.from_documents(
    documents=stage2_chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",  # save to disk
    collection_name="my_rag_docs"
)
print(f"Stored {vectorstore._collection.count()} vectors")

# ── 2b. LOAD existing vector store from disk ──────────
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
    collection_name="my_rag_docs"
)
print(f"Loaded {vectorstore._collection.count()} existing vectors")

# ── 3. Similarity search ──────────────────────────────
query = "What is retrieval augmented generation?"
results = vectorstore.similarity_search(query, k=5)
for i, doc in enumerate(results):
    print(f"\n[{i+1}] Source: {doc.metadata.get('source', 'unknown')}")
    print(f"      Text  : {doc.page_content[:150]}")

# ── 4. Search with similarity scores ─────────────────
results_scored = vectorstore.similarity_search_with_score(query, k=5)
for doc, score in results_scored:
    print(f"Score: {score:.4f} | {doc.page_content[:80]}")
# Note: Chroma returns L2 distance — LOWER score = MORE similar

# ── 5. Metadata filter ────────────────────────────────
results = vectorstore.similarity_search(
    query, k=5,
    filter={"$and": [
        {"data_type": {"$eq": "pdf"}},
        {"language": {"$eq": "en"}}
    ]}
)

# ── 6. Add more documents later ───────────────────────
new_chunks = [Document(
    page_content="New content about RAG pipelines",
    metadata={"source": "new_doc.pdf", "data_type": "pdf"}
)]
vectorstore.add_documents(new_chunks)

# ── 7. Delete documents ───────────────────────────────
# Get IDs first, then delete
collection = vectorstore._collection
results = collection.get(where={"source": "old_doc.pdf"})
if results["ids"]:
    collection.delete(ids=results["ids"])
    print(f"Deleted {len(results['ids'])} chunks from old_doc.pdf")

# ── 8. As a LangChain retriever (for chain integration) ─
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}
)
# retriever is passed directly to your LLM chain in Stage 5

# ── 9. MMR retrieval (Maximum Marginal Relevance) ─────
# Retrieves diverse results — avoids returning 5 near-identical chunks
retriever_mmr = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 5, "fetch_k": 20, "lambda_mult": 0.5}
)

faiss_complete.py — all FAISS index types and operations
from langchain_community.vectorstores import FAISS
from langchain_huggingface import HuggingFaceEmbeddings
import faiss
import numpy as np

embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-small-en-v1.5")

# ── Method 1: LangChain FAISS wrapper (easiest) ────────
vectorstore = FAISS.from_documents(stage2_chunks, embeddings)

# Search
results = vectorstore.similarity_search("What is RAG?", k=5)
results_with_scores = vectorstore.similarity_search_with_score("What is RAG?", k=5)
# FAISS returns L2 distance — lower = more similar (opposite of cosine!)

# Save and load (FAISS has NO auto-persistence)
vectorstore.save_local("./faiss_index")
vectorstore = FAISS.load_local(
    "./faiss_index", embeddings,
    allow_dangerous_deserialization=True
)

# Merge two FAISS indexes (useful for incremental indexing)
vectorstore2 = FAISS.from_documents(new_chunks, embeddings)
vectorstore.merge_from(vectorstore2)

# ── Method 2: Raw FAISS for custom index types ─────────
# Embed all chunks first
texts = [c.page_content for c in stage2_chunks]
vecs = np.array(embeddings.embed_documents(texts), dtype=np.float32)
dim = vecs.shape[1]   # e.g. 384

# Index Type A: Flat L2 (exact search, 100% recall, slow at scale)
index_flat = faiss.IndexFlatL2(dim)
index_flat.add(vecs)

# Index Type B: Flat IP (inner product / cosine for normalized vecs)
faiss.normalize_L2(vecs)    # normalize for cosine similarity
index_ip = faiss.IndexFlatIP(dim)
index_ip.add(vecs)

# Index Type C: IVFFlat (fast approximate, good for 100K+ vectors)
nlist = 100  # number of clusters (rule of thumb: sqrt(N))
quantizer = faiss.IndexFlatL2(dim)
index_ivf = faiss.IndexIVFFlat(quantizer, dim, nlist)
index_ivf.train(vecs)       # must train before adding
index_ivf.add(vecs)
index_ivf.nprobe = 10       # search 10 clusters (higher = better recall, slower)

# Index Type D: HNSW (best balance speed/recall, no training needed)
index_hnsw = faiss.IndexHNSWFlat(dim, 32)  # M=32 connections per node
index_hnsw.hnsw.efConstruction = 200
index_hnsw.add(vecs)

# Search any raw index
query_vec = np.array([embeddings.embed_query("What is RAG?")], dtype=np.float32)
faiss.normalize_L2(query_vec)
distances, indices = index_ip.search(query_vec, k=5)
for i, (idx, dist) in enumerate(zip(indices[0], distances[0])):
    print(f"[{i+1}] idx={idx} | score={dist:.4f} | {stage2_chunks[idx].page_content[:80]}")
FAISS index selection guide: <100K vectors → IndexFlatIP (exact, no training). 100K–1M vectors → IndexIVFFlat (fast ANN). 1M+ vectors → IndexIVFPQ (compressed, minimal RAM). Any size wanting easy setup → IndexHNSWFlat (no training, excellent recall).

from langchain_pinecone import PineconeVectorStore
from pinecone import Pinecone, ServerlessSpec
from langchain_huggingface import HuggingFaceEmbeddings
from dotenv import load_dotenv
import os
load_dotenv()

embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-small-en-v1.5")

# ── 1. Initialize Pinecone client ─────────────────────
pc = Pinecone(api_key=os.getenv("PINECONE_API_KEY"))

# ── 2. Create index (do once) ─────────────────────────
index_name = "rag-index"
if index_name not in [i.name for i in pc.list_indexes()]:
    pc.create_index(
        name=index_name,
        dimension=384,          # MUST match your embedding model's dim
        metric="cosine",        # or "euclidean", "dotproduct"
        spec=ServerlessSpec(
            cloud="aws",        # or "gcp", "azure"
            region="us-east-1"
        )
    )
    print(f"Index '{index_name}' created")

# ── 3. Create vector store + upsert chunks ────────────
vectorstore = PineconeVectorStore.from_documents(
    documents=stage2_chunks,
    embedding=embeddings,
    index_name=index_name
)

# ── 4. Load existing index ────────────────────────────
vectorstore = PineconeVectorStore.from_existing_index(
    index_name=index_name,
    embedding=embeddings
)

# ── 5. Similarity search ──────────────────────────────
results = vectorstore.similarity_search("What is RAG?", k=5)

# ── 6. Metadata filter ────────────────────────────────
results = vectorstore.similarity_search(
    "machine learning", k=5,
    filter={"data_type": {"$eq": "pdf"}}
)

# ── 7. Index stats ────────────────────────────────────
index = pc.Index(index_name)
stats = index.describe_index_stats()
print(f"Total vectors: {stats['total_vector_count']}")
print(f"Dimension: {stats['dimension']}")

# ── 8. As retriever ───────────────────────────────────
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
Critical: The dimension when creating a Pinecone index MUST exactly match your embedding model's output dimensions. bge-small-en-v1.5 = 384. all-MiniLM-L6-v2 = 384. bge-large = 1024. OpenAI 3-small = 1536. Mismatch → error. You cannot change dimension after creation — you'd need to delete and recreate the index.

Complete Stage 4 Pipeline END-TO-END
One script — takes Stage 3 embedding model + Stage 2 chunks → builds the vector store → exposes a retriever ready for Stage 5 (LLM generation).
This is your practice file. Set VECTOR_STORE to switch between ChromaDB, FAISS, or Qdrant. Run it after Stage 3. When you can run a similarity search and see relevant chunks returned, Stage 4 is complete.
stage4_vectorstore_pipeline.py — COMPLETE, production-ready
"""
RAG Pipeline — Stage 4: Vector Store
Input : embed_model (from Stage 3) + stage2_chunks (from Stage 2)
Output: retriever object — ready for Stage 5 (LLM chain)

VECTOR_STORE options:
  "chroma"   → ChromaDB local (recommended for learning)
  "faiss"    → FAISS in-memory + disk save
  "qdrant"   → Qdrant (Docker required)
  "pinecone" → Pinecone cloud (API key required)
"""

import os
from dotenv import load_dotenv
from langchain.schema import Document
load_dotenv()

# ─────────────────────────────────────────────────────
# CONFIGURATION — change to switch vector store
# ─────────────────────────────────────────────────────
VECTOR_STORE    = "chroma"           # "chroma" | "faiss" | "qdrant" | "pinecone"
PERSIST_DIR     = "./vectorstore"    # where to save on disk
COLLECTION_NAME = "rag_documents"    # collection / index name
TOP_K           = 5                   # number of chunks to retrieve per query


# ─────────────────────────────────────────────────────
# VECTOR STORE FACTORY
# ─────────────────────────────────────────────────────
def build_vectorstore(chunks, embed_model, store=VECTOR_STORE):
    print(f"\nBuilding {store} vector store with {len(chunks)} chunks...")

    if store == "chroma":
        from langchain_chroma import Chroma
        vs = Chroma.from_documents(
            documents=chunks, embedding=embed_model,
            persist_directory=PERSIST_DIR,
            collection_name=COLLECTION_NAME
        )
        print(f"✅ ChromaDB: {vs._collection.count()} vectors at {PERSIST_DIR}")
        return vs

    elif store == "faiss":
        from langchain_community.vectorstores import FAISS
        vs = FAISS.from_documents(chunks, embed_model)
        vs.save_local(PERSIST_DIR)
        print(f"✅ FAISS: {vs.index.ntotal} vectors saved to {PERSIST_DIR}")
        return vs

    elif store == "qdrant":
        from langchain_qdrant import QdrantVectorStore
        vs = QdrantVectorStore.from_documents(
            documents=chunks, embedding=embed_model,
            url="http://localhost:6333",
            collection_name=COLLECTION_NAME
        )
        print(f"✅ Qdrant: collection '{COLLECTION_NAME}' ready")
        return vs

    elif store == "pinecone":
        from langchain_pinecone import PineconeVectorStore
        vs = PineconeVectorStore.from_documents(
            documents=chunks, embedding=embed_model,
            index_name=COLLECTION_NAME
        )
        print(f"✅ Pinecone: index '{COLLECTION_NAME}' populated")
        return vs

    else:
        raise ValueError(f"Unknown vector store: {store}")


def load_vectorstore(embed_model, store=VECTOR_STORE):
    """Load existing vector store from disk/cloud without re-embedding."""
    print(f"Loading existing {store} vector store...")

    if store == "chroma":
        from langchain_chroma import Chroma
        return Chroma(persist_directory=PERSIST_DIR,
                       embedding_function=embed_model,
                       collection_name=COLLECTION_NAME)

    elif store == "faiss":
        from langchain_community.vectorstores import FAISS
        return FAISS.load_local(PERSIST_DIR, embed_model,
                                  allow_dangerous_deserialization=True)

    elif store == "qdrant":
        from langchain_qdrant import QdrantVectorStore
        from qdrant_client import QdrantClient
        client = QdrantClient(url="http://localhost:6333")
        return QdrantVectorStore(client=client,
                                    collection_name=COLLECTION_NAME,
                                    embedding=embed_model)
    elif store == "pinecone":
        from langchain_pinecone import PineconeVectorStore
        return PineconeVectorStore.from_existing_index(
            index_name=COLLECTION_NAME, embedding=embed_model
        )


# ─────────────────────────────────────────────────────
# MAIN PIPELINE
# ─────────────────────────────────────────────────────
def run_vectorstore(stage2_chunks, embed_model, rebuild=False):
    """
    Main Stage 4 entry point.
    rebuild=True  → rebuild the index from scratch (re-embeds everything)
    rebuild=False → load existing index (fast startup, no re-embedding)
    """
    print(f"\n{'='*54}")
    print(f" RAG Stage 4 — Vector Store ({VECTOR_STORE})")
    print(f"{'='*54}")

    if rebuild:
        vectorstore = build_vectorstore(stage2_chunks, embed_model)
    else:
        try:
            vectorstore = load_vectorstore(embed_model)
            print("✅ Loaded existing vector store")
        except:
            print("No existing store found — building from scratch")
            vectorstore = build_vectorstore(stage2_chunks, embed_model)

    # Test retrieval
    test_query = "What is the main topic of these documents?"
    results = vectorstore.similarity_search(test_query, k=3)
    print(f"\nTest query: '{test_query}'")
    print(f"Top 3 results:")
    for i, doc in enumerate(results):
        print(f"  [{i+1}] {doc.metadata.get('source','?')} | {doc.page_content[:80]}")

    # Build retriever for Stage 5
    retriever = vectorstore.as_retriever(
        search_type="mmr",        # mmr = diverse results, not just top-similar
        search_kwargs={"k": TOP_K, "fetch_k": TOP_K * 3}
    )

    print(f"\n✅ Retriever ready — passes to Stage 5 (LLM chain)")
    print(f"{'='*54}\n")

    return vectorstore, retriever


if __name__ == "__main__":
    from stage1_ingestion_pipeline import run_ingestion
    from stage2_chunking_pipeline import run_chunking
    from stage3_embedding_pipeline import run_embedding

    stage1_docs    = run_ingestion("./data")
